# Feature Specification: 비동기 예매 영속화

**Feature Branch**: `004-async-persistence`
**Created**: 2026-05-10
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 예매 성공 결과 영구 저장 (Priority: P1)

좌석 선점이 성공한 즉시 사용자에게 성공 응답이 반환되고, 예매 내역은 비동기적으로 영속 저장소에 저장된다. 사용자는 DB 저장이 완료될 때까지 기다리지 않아도 된다.

**Why this priority**: 예매 성공 이벤트가 영속화되지 않으면 시스템 재시작 시 데이터가 사라진다. 이 기능 없이는 부하 테스트의 정합성 검증이 불가능하다.

**Independent Test**: 좌석 선점 성공 후 일정 시간 내에 해당 예매 내역이 데이터베이스에 존재하는지 확인한다. 이때 예매 API의 응답 시간은 DB 저장 전후로 차이가 없어야 한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 유효한 active token으로 좌석 선점에 성공했을 때, **When** 예매 API가 성공 응답을 반환한 뒤, **Then** 해당 예매 내역(이벤트 ID, 사용자 ID, 좌석 ID, 예매 상태, 예매 시각, 멱등성 키)이 데이터베이스에 저장된다.
2. **Given** 여러 사용자가 동시에 좌석을 선점했을 때, **When** 모든 선점이 완료된 후, **Then** 데이터베이스에 저장된 예매 수는 Redis에서 성공한 선점 수와 일치한다.
3. **Given** 예매 API가 성공 응답을 반환했을 때, **When** DB 저장이 아직 진행 중이더라도, **Then** 사용자는 이미 성공 응답을 받은 상태이며 응답 지연이 없다.

---

### User Story 2 - 정합성 검증 가능 (Priority: P2)

운영자(개발자)가 Redis에서 성공한 예매 수와 데이터베이스에 저장된 예매 수가 일치하는지 쿼리로 확인할 수 있다.

**Why this priority**: 부하 테스트 후 정합성을 검증하는 것이 이 프로젝트의 핵심 검증 목표다. 조회 기능이 없으면 테스트 결과를 증명할 수 없다.

**Independent Test**: 부하 테스트 완료 후, 데이터베이스에서 이벤트별 성공 예매 수, 중복 좌석 여부, 중복 사용자 여부를 조회하여 정합성 조건을 만족하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 이벤트에 대한 예매가 완료된 상태에서, **When** 데이터베이스에서 이벤트별 성공 예매 수를 조회하면, **Then** 전체 좌석 수(2,000)를 초과하지 않는다.
2. **Given** 예매가 완료된 상태에서, **When** 동일 좌석의 예매 내역을 조회하면, **Then** 하나의 사용자만 존재한다.
3. **Given** 예매가 완료된 상태에서, **When** 동일 사용자의 예매 내역을 조회하면, **Then** 동일 이벤트에 대해 하나의 예매만 존재한다.

---

### User Story 3 - Worker 중단 시 이벤트 유실 없음 (Priority: P3)

영속화 워커가 일시적으로 중단되거나 데이터베이스 연결이 끊겨도 예매 성공 이벤트가 유실되지 않는다. 워커가 재시작되면 처리되지 않은 이벤트를 이어서 처리한다.

**Why this priority**: 데이터 유실은 정합성 원칙 위반이다. 로컬 실험 환경에서도 워커 재시작 시나리오를 검증해야 한다.

**Independent Test**: 워커를 강제 중단 후 재시작했을 때, 중단 기간 동안 발생한 예매 성공 이벤트가 모두 데이터베이스에 저장되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 예매 성공 이벤트가 발생한 직후 워커가 중단될 때, **When** 워커가 재시작되면, **Then** 중단 기간 동안 처리되지 않은 이벤트가 데이터베이스에 저장된다.
2. **Given** 데이터베이스 연결이 일시적으로 끊겼을 때, **When** 연결이 복구되면, **Then** 실패한 이벤트가 재처리되어 데이터베이스에 저장된다.

---

### Edge Cases

- 동일 예매 이벤트가 중복으로 저장 시도될 때 중복 저장을 방지해야 한다.
- 워커가 이벤트를 소비하던 중 DB 저장에 실패하면 이벤트를 재처리 가능한 상태로 유지해야 한다.
- 좌석 선점은 성공했지만 이벤트 발행에 실패하는 경우 시스템 동작을 정의해야 한다.
- 대량의 이벤트가 동시에 발생할 때 워커가 처리 속도를 따라가지 못하는 경우(worker lag)를 관찰할 수 있어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 좌석 선점 성공 시 예매 성공 이벤트를 영속화 대기열에 등록해야 한다.
- **FR-002**: 예매 API는 이벤트 대기열 등록 후 즉시 응답을 반환해야 하며, 데이터베이스 저장 완료를 기다려서는 안 된다.
- **FR-003**: 영속화 워커는 이벤트를 소비하여 예매 내역을 데이터베이스에 저장해야 한다.
- **FR-004**: 저장할 예매 내역은 예매 ID, 이벤트 ID, 사용자 ID, 좌석 ID, 예매 상태, 예매 시각, 멱등성 키를 포함해야 한다.
- **FR-005**: 동일 이벤트가 중복 저장 시도되는 경우 데이터베이스에 중복 저장되어서는 안 된다.
- **FR-006**: 워커가 재시작된 후 처리되지 않은 이벤트를 이어서 소비할 수 있어야 한다.
- **FR-007**: 이벤트 대기열은 데이터베이스를 사용하지 않아야 한다.
- **FR-008**: 부하 테스트 후 이벤트 ID별 성공 예매 수, 중복 좌석, 중복 사용자를 쿼리로 검증할 수 있어야 한다.

### Key Entities

- **예매 이벤트(Reservation Event)**: 좌석 선점 성공 시 발행되는 이벤트. 예매 ID, 이벤트 ID, 사용자 ID, 좌석 ID, 예매 상태, 예매 시각, 멱등성 키를 포함한다.
- **예매 내역(Reservation Record)**: 데이터베이스에 영구 저장되는 예매 결과. 예매 이벤트로부터 생성된다.
- **영속화 워커(Persistence Worker)**: 이벤트 대기열을 소비하여 예매 내역을 데이터베이스에 저장하는 처리기.
- **이벤트 대기열(Event Queue)**: 예매 성공 이벤트를 임시 보관하는 대기열. 워커가 처리할 때까지 이벤트를 보존한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 좌석 선점 성공 후 데이터베이스에 저장되기까지 걸리는 최대 지연 시간이 부하 테스트 환경에서 관찰되고 기록된다.
- **SC-002**: 부하 테스트 완료 후 데이터베이스에 저장된 성공 예매 수는 Redis에서 성공한 선점 수와 100% 일치한다.
- **SC-003**: 데이터베이스에 동일 좌석이 두 개 이상 저장된 행이 존재하지 않는다.
- **SC-004**: 데이터베이스에 동일 사용자가 동일 이벤트에 두 개 이상 저장된 행이 존재하지 않는다.
- **SC-005**: 좌석 선점 API의 응답 시간은 영속화 워커 도입 전후로 차이가 없다.
- **SC-006**: 워커 재시작 후 미처리 이벤트가 모두 데이터베이스에 저장된다.

## Assumptions

- 이벤트 대기열은 애플리케이션 내부 또는 Redis 기반으로 구현하며, 외부 메시지 브로커(Kafka, RabbitMQ)는 이 기능 범위에 포함하지 않는다.
- 예매 ID는 시스템이 생성하는 고유 식별자이며 클라이언트가 제공하지 않는다.
- 데이터베이스 스키마는 이 기능에서 처음 정의된다(기존 MySQL 테이블 없음).
- 워커는 단일 인스턴스로 시작하며, 다중 워커 인스턴스 간 동시성 문제는 이 기능 범위에 포함하지 않는다.
- Docker Compose로 MySQL이 로컬에서 실행 가능해야 한다.
- 이벤트 대기열이 가득 찬 경우(용량 초과)의 처리는 이 기능 범위에 포함하지 않는다.
